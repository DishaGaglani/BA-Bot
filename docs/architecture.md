# Project State Engine Architecture

This document describes the structured **Project State Engine** and **State-Driven Elicitation Manager** architecture implemented inside BA-Bot.

---

## 1. Architectural Goals
- **Minimize AI Context overhead**: Shift prompt generation from raw, verbose dialogue histories to structured project state vectors.
- **Resilience & Continuity**: Keep raw conversation history secondary. Only include a sliding window of the last 2 turns inside LLM prompts to maintain immediate dialog flow.
- **Targeted Updates**: Ensure user replies dynamically update only the active requirement stage sections.

---

## 2. Project State Properties
Every project workspace maintains a structured state (persisted as serialized JSON inside the SQLite database) containing:

| State Key | Description |
|---|---|
| `project_name` | Name of the project |
| `industry` | Project domain or vertical (Business Objectives) |
| `department` | Sponsoring department |
| `sponsor` | Primary sponsor |
| `business_unit` | Specific business unit target |
| `timeline` | Project timeline metrics (Assumptions) |
| `budget` | Allocation budget (Constraints) |
| `business_requirements` | Extracted business requirements |
| `functional_requirements` | List of functional features (`{"title", "priority", "confidence"}`) |
| `non_functional_requirements`| List of performance/security/availability constraints |
| `stakeholders` | List of project stakeholders |
| `constraints` | Core business/timeline limitations |
| `assumptions` | Primary resource or technical assumptions |
| `risks` | Key security or project delivery risks |
| `integrations` | Required third-party system connections |
| `user_roles` | Targeted end-user roles |
| `asked_questions` | Persistent log of specific question sentences presented by the AI |
| `pending_questions` | Tracker for active workshop checklist questions |
| `completed_sections` | Checklist of completed discovery segments |
| `generated_summaries` | Exec-level summary string generated from active state |

## 3. State-Driven Elicitation Manager

### 3.1 Dependency Graph & Priority
The state controller in [gap_analyzer.py](file:///Users/dishagaglani/Desktop/L&T%20PES/BA_BOT/ba-agent/backend/services/gap_analyzer.py) enforces pre-requisites before any requirement section becomes targetable:
- `functional_requirements` depends on `industry` (Business Objectives)
- `non_functional_requirements` depends on `functional_requirements`
- `integrations` depends on `functional_requirements`
- `constraints` depends on `timeline` and `budget`

If a section's pre-requisites are unfulfilled, it is automatically skipped for active elicitation, ensuring structured question progression.

### 3.2 Answered Question & Duplicate Prevention
- **Dynamic Skipping**: If a user's response populates multiple sections (e.g. combined inputs), they are immediately parsed by the delta extractor and marked complete in the Project State. The gap analyzer automatically skips completed items, preventing redundant questioning.
- **Asked Question Logger**: All question sentences generated by the AI are extracted (`re.findall(r'([^.!?]*\?)', ai_reply)`) and logged inside `asked_questions`. This list is passed to the LLM prompt to prevent duplicate or repeated questions.

## 4. Prompt Context Minimization
- The system prompt generator in [prompt_builder.py](file:///Users/dishagaglani/Desktop/L&T%20PES/BA_BOT/ba-agent/backend/services/prompt_builder.py) loads:
  1. System Prompt
  2. Project Metadata
  3. Current Requirement Stage Focus
  4. State-derived Generated Summary
  5. **Recent Messages**: Limited to the last 2 turns.
  6. Elicitation prompt configurations & validation rules.
  7. **Asked Questions**: Last 5 logged questions to avoid duplication.
- Old conversation history is completely ignored in the prompt window, preventing context token bloating and reducing AI transaction latency/costs.
